# AWS Management Console Access & Navigation Guide

**For:** Students/Learers, New Cloud Engineers | **Applies to:** Dev & Production | Windows, macOS, Linux | **Example Region:** us-east-1

---

## 1. What is the AWS Management Console?

The AWS Management Console is your web based dashboard for working with AWS services. Think of it as AWS's control room; you'll use it to launch servers, manage storage, configure permissions, and more.

**Why secure access matters:**
Getting access wrong can expose sensitive data, result in unexpected charges, or create compliance violations. Setting things up correctly from day one is worth the extra effort.

---

## 2. Before You Begin: What You'll Need

- **AWS account info:**
  - Account ID & name
  - Organization/OU (if applicable)
- **How you'll log in:**
  - Production: SSO (Single Sign On) portal
  - Development: IAM user credentials (though IAM Identity Center is recommended even here; see Section 4)
- **Security device:**
  - MFA app (Oracle Mobile Authenticator, Google Authenticator, Authy, Microsoft Authenticator) or a hardware/passkey device
    
>**Note:** AWS now recommends phishing resistant MFA; passkeys or FIDO2 security keys (e.g., YubiKey); as the preferred option over TOTP apps, which remain valid but offer weaker protection.

- **Your user provisioned:**
  - Either an IAM or IAM Identity Center (SSO) user, with appropriate roles assigned

## 4. Setting Up Access

### 4.1 Create an AWS Account

1. Go to [aws.amazon.com](https://aws.amazon.com/) > **Create an AWS Account**
2. Enter your details, organization info, and payment method.
3. Verify via phone or email.
4. Choose a support plan (start with **Basic**; you can upgrade later).
5. Set a strong root password and **enable MFA immediately**.

> **Important Note:** Never use the root account for everyday tasks. Create an administrator IAM user or set up IAM Identity Center right away.

---

### 4.2 User Setup & Login

#### A. Developer Access (IAM User)

While IAM users work on development, consider using IAM Identity Center (Section 4.2B) even here to avoid managing long term credentials.

1. Log in as root at [AWS Console](https://console.aws.amazon.com/) (for initial setup only).
2. Navigate to: `Services` > `IAM` > `Users` > **Add user**
3. Username: `<YOUR_USERNAME>`
4. Enable Console access and set a temporary password (require reset on first login).
5. Add the user to a group that has the appropriate policy attached (e.g., `ReadOnlyAccess`).
6. Save credentials securely using a password manager.
7. Log out and log back in as the new IAM user.

> **Important Note:** Attach permission policies to groups, then add users to those groups; do not attach policies directly to individual users. The CLI example below reflects this.

**Alternatively**
**Using CLI:**
- Create the user
```bash
aws iam create-user --user-name <YOUR_USERNAME>
```

- Create a group and attach the policy to the group
```bash
aws iam create-group --group-name ReadOnly
aws iam attach-group-policy --group-name ReadOnly --policy-arn arn:aws:iam::aws:policy/ReadOnlyAccess
```

- Add the user to the group
```bash
aws iam add-user-to-group --user-name <YOUR_USERNAME> --group-name ReadOnly
```

#### B. Production Access (IAM Identity Center / SSO)

AWS IAM Identity Center replaced AWS Single Sign On (AWS SSO) in 2022 and is now the recommended way to manage human access across all environments.

1. Set up [IAM Identity Center](https://docs.aws.amazon.com/singlesignon/latest/userguide/getting-started.html) from your management account.
2. Choose your identity source: built-in directory, Microsoft Active Directory, or an external IdP (Okta, Microsoft Entra ID, Google Workspace, etc.).
3. For existing IAM users in production, disable their console login:
   `IAM` > `Users` > `<username>` > `Security credentials` > **Deactivate Console access**
4. Confirm that only IAM Identity Center users have console access going forward.

---

### 4.3 Setting Up MFA

- **IAM Users:**
  `IAM` > `Users` > `<username>` > `Security credentials` > **Assign MFA device**
- **IAM Identity Center:**
  Enforce MFA through your identity provider's settings, or via IAM Identity Center's built-in MFA configuration.
- Scan the QR code using your chosen MFA app or enroll your security key, then enter two consecutive codes to complete setup.

> AWS recommends phishing resistant MFA such as passkeys and security keys wherever possible. These FIDO (Fast Identity Online) based authenticators use public key cryptography and are resistant to phishing, man in the middle, and replay attacks; providing stronger protection than TOTP (Time based One Time Password) based options. TOTP apps (Oracle Mobile Authenticator, Google Authenticator, Authy, Microsoft Authenticator) remain a valid interim option.

### Explore the Console

1. Log in with your IAM or IAM Identity Center credentials.
2. Use the **Search Bar** to find a service (e.g., "S3").
3. Pin frequently used services for quick access.
4. Use the **Account Menu** for role switching and account settings.
5. Visit the **Billing Dashboard** to track costs and set up alerts.
6. Explore the **Resource Center** for tutorials and AWS news.

---

### Role Switching

1. Click your account name (top right) > **Switch Role**
2. Enter:
   - Account ID: `<YOUR_AWS_ACCOUNT_ID>`
   - Role name: `<ROLE_NAME>`
   - Display name/color (optional, for visual clarity)
3. Click **Switch Role**
4. Confirm the active role shown at the top of the console.

## 6. Verification Steps

- Confirm your username and active role appear in the top right corner.
- Try accessing a resource (e.g., an S3 bucket) to validate your permissions.
- Verify that login prompts for MFA.
- Check AWS CloudTrail for console login events to confirm audit logging is working.

---

## 7. Problem and Solutions

- **Permission Denied:** Check your group or role assignment and validate the attached policy in IAM.
- **MFA Issues:** Re sync the device clock or re enroll the MFA device in IAM.
- **Role Switch Error:** Verify the role's trust policy and that your user is listed as a trusted principal.
- **Console Access Disabled:** Review IAM Identity Center settings and confirm you're logging in through the correct portal.

---

## 8. Cleaning Up

- Remove IAM users and roles that are no longer needed.
- Detach any permissions that are no longer required.
- Delete test resources to avoid ongoing charges.
- Review tags and the Billing Dashboard to identify "orphaned" resources.

**Using CLI:**
```bash
aws iam list-users
aws iam delete-user --user-name <USERNAME>
```

---

## 9. Cost Management

- IAM users, groups, and roles are free of charge.
- You are billed only for the AWS resources you create and use (EC2 instances, S3 storage, etc.).
- Use the **Billing Dashboard** and **Cost Explorer** (`Console` > `Billing`) to track spending and identify unexpected costs.

---

## 10. Recommendations

- Use IAM Identity Center for all human access, including development.
- Disable console access for IAM users in production environments.
- Enforce MFA across all users, and prefer phishing resistant options like passkeys and security keys.
- Regularly audit logins, permissions, and resource usage.
- Tag all resources with project and team identifiers for cost tracking.
- Set billing alerts and budget thresholds to catch runaway costs early.
- Always use groups to assign permissions; this ensures consistency and reduces the chance of misconfiguration.

---

## 11. Minimal IAM Policy Example

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": "s3:ListAllMyBuckets",
    "Resource": "*"
  }]
}
```
*Allows listing all S3 buckets in the account. In production, restrict permissions to only what's needed for the specific task.*
