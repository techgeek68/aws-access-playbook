
# Lab: Creating Security Groups in AWS Management Console

This guide walks through creating Security Groups in AWS using the Management Console. Security Groups act as virtual firewalls, controlling inbound and outbound network traffic for your EC2 instances.

---

## What is a Security Group?

A **Security Group** is a collection of rules that govern network traffic to and from AWS resources, primarily EC2 instances. You define which IP ranges, protocols, and ports are permitted for both incoming and outgoing connections. Each security group is tied to a specific VPC and can only be used within that VPC.

---

## Prerequisites

- AWS account with permissions to manage EC2 resources.
- Web browser.

---

## Procedure

### 1. Sign in to AWS Management Console

1. Go to [https://aws.amazon.com/](https://aws.amazon.com/).
2. Click **Sign In to the Console**.
3. Enter your credentials and complete MFA if prompted.

---

### 2. Navigate to EC2 Security Groups

1. From the AWS Console home page, type **EC2** in the search bar and select **EC2**.
2. In the left sidebar, find the **Network & Security** section.
3. Click **Security Groups**.

---

### 3. Create a New Security Group

1. Click **Create security group** in the top right corner of the page.
2. Fill in the following fields:
   - **Security group name**: Choose a clear, descriptive name (e.g., `MyLabSG`).
>**Note:** The name and description cannot be changed after the security group is created.
   - **Description**: Briefly state the group's purpose (e.g., `Allow SSH and HTTP access for lab`).
   - **VPC**: Select the VPC to associate this security group with. The security group can only be used within this VPC.

---

### 4. Add Inbound Rules

**Inbound rules** control traffic coming **into** your resources.

1. Click **Add rule**.
2. Configure each rule with the following:
   - **Type**: Choose the protocol or service (e.g., SSH, RDP, HTTP).
   - **Protocol**: Auto fills based on the selected type.
   - **Port range**: Auto fills for standard types; enter manually for custom ports.
   - **Source**: Use `My IP` for SSH/RDP access, or enter a specific IP/CIDR range.
     - For public web traffic, `0.0.0.0/0` (IPv4) and `::/0` (IPv6) allow access from all addresses. Reserve this for HTTP/HTTPS only, and avoid it for SSH and RDP.
3. Common example rules:
   - SSH: Type=`SSH`, Port=`22`, Source=`My IP`
   - RDP: Type=`RDP`, Port=`3389`, Source=`My IP`
   - HTTP: Type=`HTTP`, Port=`80`, Source=`0.0.0.0/0` (and `::/0` for IPv6)
   - HTTPS: Type=`HTTPS`, Port=`443`, Source=`0.0.0.0/0` (and `::/0` for IPv6)
4. Add as many rules as needed.

---

### 5. Add Outbound Rules (Optional)

**Outbound rules** control traffic going **out** from your resources.

- By default, new security groups include an outbound rule that allows all traffic to leave. You only need to modify this if your use case requires restricting egress traffic.
- To restrict outbound traffic, click **Add rule** and specify the destination, protocol, and port.

### 7. Attach the Security Group to an EC2 Instance

- **New instance**: In the launch wizard, find the **Network settings** section and select your security group there.
- **Existing instance**:
  1. Go to **Instances** in the EC2 Dashboard.
  2. Select your instance.
  3. Click **Actions** > **Security** > **Change security groups**.
  4. Add or remove security groups as needed and save.

---

## Recommendations

- **Restrict access**: Lock SSH and RDP to your specific IP — never open them to `0.0.0.0/0` in production.
- **Naming**: Choose names that reflect the role, environment, or team (e.g., `prod-web-sg`). Remember, names cannot be renamed after creation.
- **Least privilege**: Only open ports your application actually requires.
- **IPv6 awareness**: If your VPC supports IPv6, add `::/0` alongside `0.0.0.0/0` for public facing rules.
- **Regular audits**: Review security groups periodically and remove stale or overly permissive rules.

---

## Cleanup

Remove security groups that are no longer in use. Note that you cannot delete a security group that is currently attached to a running instance; detach it first.

1. Select the security group from the list.
2. Click **Actions** > **Delete security groups**.
3. Confirm the deletion.
