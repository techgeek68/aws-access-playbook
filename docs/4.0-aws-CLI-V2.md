# AWS CLI v2: Installation, Configuration, MFA & Troubleshooting

---

## 1. Overview
The AWS CLI provides a single, scriptable interface for interacting with AWS services from the command line. This guide covers the core CLI lifecycle: 

   `install > configure > authenticate (including MFA and role assumption) > verify > troubleshoot`


---

## 2. Supported Platforms
   - **Linux** (x86_64, aarch64): Ubuntu/Debian, RHEL/CentOS, Amazon Linux
   - **macOS**: Intel and Apple Silicon (M-series)
   - **Windows**: 10 / 11, 64-bit (PowerShell and Command Prompt)

---

## 3. Prerequisites
- An AWS account with IAM user, federated SSO, or role based access
- Administrative or `sudo` rights for installation
- `curl` and `unzip` (Linux/macOS manual install)
- `jq` (optional but handy for parsing JSON in shell scripts)
- A terminal: `bash`, `zsh`, or `PowerShell`

---

## 4. Installation

> Only follow the section that matches your operating system.


### 4.1 Linux (Ubuntu/Debian/RHEL/CentOS/Amazon Linux)

**Update packages:**

- Debian / Ubuntu:
```bash
sudo apt update && sudo apt upgrade -y
```

- RHEL / CentOS / Amazon Linux:
```bash
sudo yum update -y
```

**Install prerequisites:**
- Debian/Ubuntu
```bash
sudo apt install -y curl unzip
```
- RHEL/CentOS/Amazon Linux
```bash
sudo yum install -y curl unzip
```

**Check your architecture:**
```bash
uname -m
```
>Common outputs: `x86_64`, `aarch64`

**Download (pick the right architecture):**

x86_64:
```bash
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
```

ARM (aarch64):
```bash
curl "https://awscli.amazonaws.com/awscli-exe-linux-aarch64.zip" -o "awscliv2.zip"
```

**Unzip and install:**
```bash
unzip awscliv2.zip
```
```bash
sudo ./aws/install
```

>Custom install location:
>By default, files go to `/usr/local/aws-cli` and a symlink is created in `/usr/local/bin`. To specify custom paths:
>```bash
>sudo ./aws/install -i /usr/local/aws-cli -b /usr/local/bin
>```

**Verify:**
```bash
aws --version
```

---

If `aws` is not recognized after installation:
- Close all open terminal and PowerShell windows — PATH changes don't apply to already-open shells.
- The default installation directory for AWS CLI v2 on Windows is:
  ```
  C:\Program Files\Amazon\AWSCLIV2\
  ```
  ⚠️ **Correction from original doc:** There is no `bin` subdirectory — `aws.exe` lives directly inside `AWSCLIV2\`, not in a `bin` subfolder.
- To add it to PATH manually:
  - Press **Win + R**, type `sysdm.cpl`, press **Enter**.
  - Go to **Advanced** > **Environment Variables**.
  - Under **System variables**, select **Path** > click **Edit**.
  - Click **New** and paste:
    ```
    C:\Program Files\Amazon\AWSCLIV2\
    ```
  - Click **OK** on all dialogs, then restart your terminal.

**Uninstall:**
```powershell
winget uninstall Amazon.AWSCLI
```
Or use **Control Panel > Programs > Uninstall a program** (for MSI installs).

## 6. Configuration

AWS CLI stores credentials and settings in two files:
- **Credentials:** `~/.aws/credentials` (Linux/macOS) or `%UserProfile%\.aws\credentials` (Windows)
- **Config:** `~/.aws/config` (Linux/macOS) or `%UserProfile%\.aws\config` (Windows)

---

### 6.1 Interactive Default Profile

```bash
aws configure
```

You'll be prompted for:
- AWS Access Key ID
- AWS Secret Access Key
- Default region (e.g., `us-east-1`)
- Default output format (`json`, `yaml`, `text`, or `table`)

> **Note:** If your Access Key ID starts with `ASIA`, you have temporary STS credentials. You'll also need to add the session token manually.

Add session token after running `aws configure`:

**Linux/macOS:**
```bash
vi ~/.aws/credentials
```

**Windows:**
```powershell
notepad $env:UserProfile\.aws\credentials
```
Add this line under your profile: `aws_session_token = <SESSION_TOKEN>`

Verify:
```bash
aws sts get-caller-identity
```

---

### 6.2 Credentials & Config File Examples

Linux/macOS: `vim ~/.aws/credentials`  
Windows: `notepad $env:UserProfile\.aws\credentials`

`~/.aws/credentials`:
```ini
[default]
aws_access_key_id = AKIA_DEFAULT...
aws_secret_access_key = defaultSecret...

[prod]
aws_access_key_id = AKIA_PROD...
aws_secret_access_key = prodSecret...
```

`~/.aws/config`:
```ini
[default]
region = us-east-1
output = json

[profile prod]
region = us-west-2
output = json
```

> Note: Named profiles in `~/.aws/config` require the `profile` prefix (e.g., `[profile prod]`), while in `~/.aws/credentials` you just use `[prod]`.

---

### 6.3 Named Profiles

Create a named profile:
```bash
aws configure --profile prod
```

Use a specific profile for a single command:
```bash
aws s3 ls --profile prod
```

Set a profile as default for your entire shell session:

Linux/macOS:
```bash
export AWS_PROFILE=prod
```

PowerShell:
```powershell
$Env:AWS_PROFILE = 'prod'
```

---

### 6.4 Environment Variables (Override Mechanism)

Environment variables let you override profile settings without touching `~/.aws/credentials`. They take the highest precedence in the credential resolution chain.

Bash:
```bash
export AWS_ACCESS_KEY_ID="AKIA..."
export AWS_SECRET_ACCESS_KEY="..."
export AWS_SESSION_TOKEN="..."         # only for temporary credentials
export AWS_DEFAULT_REGION="us-east-1"
```

PowerShell:
```powershell
$Env:AWS_ACCESS_KEY_ID     = "AKIA..."
$Env:AWS_SECRET_ACCESS_KEY = "..."
$Env:AWS_SESSION_TOKEN     = "..."
$Env:AWS_DEFAULT_REGION    = "us-east-1"
```

Unset variables:

Linux/macOS:
```bash
unset AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN AWS_PROFILE
```

PowerShell:
```powershell
"ACCESS_KEY_ID","SECRET_ACCESS_KEY","SESSION_TOKEN","PROFILE","DEFAULT_REGION" | ForEach-Object { Remove-Item Env:AWS_$_ -ErrorAction SilentlyContinue }
```

---

### 6.5 Profiles vs Environment Variables

| Aspect | Profiles | Environment Variables |
|---|---|---|
| Persistence | Stored on disk | Session-scoped only |
| Typical Use | Day-to-day dev contexts | CI/CD, ephemeral STS sessions |
| Priority | Lower (overridden by env vars) | Higher (overrides profiles) |
| Key Rotation | Manual file edit | Easy to swap frequently |

**Recommendation:** Use profiles for stable, recurring contexts. Use environment variables in automation pipelines or when working with temporary STS credentials.

## 7. MFA & Temporary Credentials (STS)

### 7.1 get-session-token (Stay as IAM User)

Use this when your IAM policies require MFA but you don't need to change your effective identity.

Request:
```bash
aws sts get-session-token \
  --serial-number arn:aws:iam::123456789012:mfa/your-username \
  --token-code 123456 \
  --duration-seconds 3600
```

Response (truncated):
```json
{
  "Credentials": {
    "AccessKeyId": "ASIA....",
    "SecretAccessKey": "abcd...",
    "SessionToken": "IQoJb3...",
    "Expiration": "2025-08-17T12:34:56Z"
  }
}
```

Export credentials (bash):
```bash
export AWS_ACCESS_KEY_ID="ASIA..."
export AWS_SECRET_ACCESS_KEY="abcd..."
export AWS_SESSION_TOKEN="IQoJb3..."
```

Or save to a named profile:
```bash
aws configure set aws_access_key_id     "ASIA..."    --profile mfa
aws configure set aws_secret_access_key "abcd..."    --profile mfa
aws configure set aws_session_token     "IQoJb3..."  --profile mfa
```

Use:
```bash
aws s3 ls --profile mfa
```

---

### 7.2 assume-role (Change Effective Identity)

Used for cross-account access, privilege escalation with MFA, or duty separation.

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/AdminRole \
  --role-session-name mySession \
  --serial-number arn:aws:iam::123456789012:mfa/your-username \
  --token-code 123456
```

Extract and export credentials (bash with `jq`):
```bash
CREDS_JSON=$(aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/AdminRole \
  --role-session-name mySession)

export AWS_ACCESS_KEY_ID=$(echo "$CREDS_JSON"     | jq -r '.Credentials.AccessKeyId')
export AWS_SECRET_ACCESS_KEY=$(echo "$CREDS_JSON" | jq -r '.Credentials.SecretAccessKey')
export AWS_SESSION_TOKEN=$(echo "$CREDS_JSON"     | jq -r '.Credentials.SessionToken')
```

PowerShell:
```powershell
$creds = aws sts assume-role `
  --role-arn arn:aws:iam::123456789012:role/AdminRole `
  --role-session-name mySession | ConvertFrom-Json

$Env:AWS_ACCESS_KEY_ID     = $creds.Credentials.AccessKeyId
$Env:AWS_SECRET_ACCESS_KEY = $creds.Credentials.SecretAccessKey
$Env:AWS_SESSION_TOKEN     = $creds.Credentials.SessionToken
```

---

### 7.3 Comparison

| Feature | get-session-token | assume-role |
|---|---|---|
| Identity | Same IAM user | Role (different principal) |
| Permissions | Same as the IAM user | Defined by the role's policy |
| Typical Use | Add MFA enforcement to user | Cross-account / elevated rights |
| Max Duration | Up to 36 hours (IAM user policy dependent) | Up to 12 hours (role's MaxSessionDuration setting) |
| MFA | At call time, if required | Enforced via role trust policy |
| ExternalId | Not applicable | Supported |

## 8. CLI Troubleshooting

| Symptom | Likely Cause | Resolution |
|---|---|---|
| `aws: command not found` | PATH not set | Reinstall or manually add the install `bin` path to PATH |
| `'aws' is not recognized` (Windows) | PATH not refreshed | Reopen terminal; or add `C:\Program Files\Amazon\AWSCLIV2\` to PATH |
| `AccessDenied` | Missing IAM permissions | Check active identity with `aws sts get-caller-identity` |
| Invalid MFA token | Clock drift or wrong device | Check device time sync and verify the MFA ARN |
| `Could not connect to the endpoint URL` | Wrong region, network issue, or proxy | Set `--region` flag or `AWS_DEFAULT_REGION` env var |
| SSL errors | Corporate MITM proxy / missing CA bundle | Set `AWS_CA_BUNDLE` to your CA path or configure proxy trust |
| Wrong profile being used | Env var taking precedence | Run `unset AWS_PROFILE`, or inspect with `env \| grep AWS` |
| `ExpiredToken` | STS credentials have expired | Re-run `get-session-token` or `assume-role` |

Inspect your current identity:
```bash
aws sts get-caller-identity
```

Inspect credential files:
```bash
cat ~/.aws/credentials
cat ~/.aws/config
```

## 9. Best Practices

- Prefer **roles with MFA** over long-lived IAM user access keys.
- Apply least privilege: use separate read-only, dev, and prod profiles.
- Never hardcode credentials in scripts; inject them via environment variables or a secrets manager in CI/CD.
- Rotate access keys regularly and delete any that are unused.
- Specify `--profile` explicitly in automation scripts to avoid ambiguity.
- Use `--dry-run` where supported (e.g., certain EC2 commands) to validate commands before they run.
- Use `--output yaml` or `--query` for cleaner, structured output in scripts.
- Use `aws configure sso` if your organization uses AWS IAM Identity Center (formerly AWS SSO).
