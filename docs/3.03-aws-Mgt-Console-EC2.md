
# Lab: Creating an Amazon EC2 Instance Using the AWS Management Console

A practical walkthrough for launching and connecting to both Linux and Windows EC2 instances through the AWS Management Console. Each AWS concept is explained where it's introduced so you understand what you're configuring and why.

---

## Step 1: Sign Into AWS Management Console

1. Go to [https://aws.amazon.com/](https://aws.amazon.com/).
2. Click **Sign In to the Console**.
3. Enter your credentials; either an IAM user (common for development and learning) or an SSO user (common in organizational/production environments).
4. If MFA is enabled on your account, enter your one time code when prompted.
5. Click **Sign In**.
6. Confirm you're working in the right AWS region using the dropdown in the top right corner (e.g., `us-east-1`). Resources are region specific, so this matters.

---

## Step 2: Navigate to EC2 Dashboard

1. Type `EC2` into the search bar at the top of the AWS Console.
2. Select **EC2** from the results.
3. Double check the region shown in the top right; make sure it matches your intended region before proceeding.

## Step 3: Open the Launch Instance Wizard

1. On the EC2 Dashboard, click the orange **Launch Instance** button.
2. This opens the redesigned launch wizard, which AWS rebuilt as a **single page experience**; all configuration sections are presented together on one screen rather than across separate steps. You'll work through each section on this page and launch when ready.



---

## Step 4: Add Tags (Optional)

**Tags:** Key value labels that help you identify, organize, and track costs for your AWS resources.

1. In the **Name and tags** section (at the top of the wizard), your instance name is already stored as a `Name` tag.
2. To add more tags, click **Add additional tags** and enter a **Key** (e.g., `Environment`) and **Value** (e.g., `Dev`).
3. Useful tags include owner, project, and cost center; especially helpful in shared or billing tracked accounts.
4. Example:
   - Name: `My Web Server`

---

## Step 5: Choose an Amazon Machine Image (AMI)

**What is an AMI?**
An **Amazon Machine Image** is a pre configured template that defines the operating system and software stack your instance starts with. Think of it as a snapshot that EC2 uses to boot your virtual machine.

1. Under **Application and OS Images (Amazon Machine Image)**, browse or search for your preferred OS; common choices include **Amazon Linux**, **Red Hat**, **Ubuntu**, and **Windows Server**.
2. Click **Select** next to the AMI you want.
3. Review the architecture (x86\_64 or ARM), storage requirements, and support level before confirming.
4. Example:
   - Search: `Red Hat`
   - 64-bit (x86)
   - Select
     
---

## Step 6: Choose an Instance Type

**What is an Instance Type?**
The **instance type** determines the hardware your instance runs on — how many virtual CPUs it gets, how much RAM, and its networking capacity.

1. Review the available types. For Free Tier eligibility, select `t2.micro` (or `t3.micro` in regions where `t2.micro` is not available).
2. Compare the vCPU count, memory, and estimated cost for your workload.
3. Select the instance type that fits your needs.

## Step 7: Create or Select a Key Pair

**Key Pair:** A public/private key set used to authenticate your connection to the instance. AWS stores the public key; you keep the private key.

1. Under **Key pair (login)**, choose an existing key pair or create a new one.
2. To create a new key pair:
   - Enter a unique name.
   - Choose your format:
     - `.pem` — works natively on Linux, macOS, and **Windows 10/11** (which includes built-in OpenSSH).
     - `.ppk` — required only if you're using **PuTTY** on Windows.
   - Click **Create key pair** to download the file.
3. Store the private key in a secure location. **You cannot download it again after this point.**
4. Example:
   - Key pair (login): Key pair name > Loginkeyforcloud

---

## Step 8: Configure Instance Details

**VPC (Virtual Private Cloud):** A logically isolated network environment within AWS where your resources live.

**Subnet:** A segment of your VPC tied to a specific availability zone, defining the IP range your instance draws from.

**IAM Role:** Lets your instance authenticate to other AWS services (like S3 or SSM) without hardcoding credentials.

In the **Network settings** section of the wizard:

1. Choose a **VPC** and **Subnet** (availability zone). The defaults work fine for most beginners.
2. Enable **Auto assign Public IP** if you need to connect to the instance directly from the internet.
3. Optionally attach an **IAM role** if the instance needs to call other AWS services.
4. Enable **Detailed Monitoring** under Advanced settings if you want finer-grained CloudWatch metrics (this incurs additional cost).
5. Example:
   - VPC: Choose default
   - Subnet: Choose any e.g.

   - Auto-assign public IP: Enable

**Security Group:** A virtual firewall that controls what traffic is allowed in and out of your instance.

1. Under **Network settings**, choose **Create security group** or select an existing one.
2. Add inbound rules based on your instance's OS:
   - **Linux (SSH):** Type = `SSH`, Protocol = `TCP`, Port = `22`, Source = `My IP`
   - **Windows (RDP):** Type = `RDP`, Protocol = `TCP`, Port = `3389`, Source = `My IP`
   - Add `HTTP` (port 80) or `HTTPS` (port 443) only if you're hosting a web server.
3. Give your security group a descriptive name.
4. Example:
      - Firewall (security groups): Select existing security group > Common security groups
 Info (My_WebServer_SecurityGroup)

> **Note:** Always restrict source access to **My IP** rather than `0.0.0.0/0` (which exposes the port to the entire internet). This is especially important for SSH and RDP, which are common targets for automated brute force attacks.

 
---

## Step 9: Add Storage (EBS)

**EBS (Elastic Block Store):** Persistent block storage volumes that attach to your EC2 instance, similar to a hard drive.

1. The wizard pre fills a root volume based on the AMI you selected.
2. The default is an 8 GB `gp3` root volume. `gp3` is the current general purpose SSD type and is the default for new volumes. You can increase the size if needed.
3. To add a second volume, click **Add new volume** and configure its type (`gp3`, `io2`, etc.), size, and encryption settings.
4. The Free Tier includes up to 30 GB of EBS storage, so you can increase your root volume size if needed without extra cost.
5. Example:
   - 10 GiB   gp3

---

## Step 10: Launch the Instance

1. Review the **Summary** panel on the right side of the wizard; it shows your selected AMI, instance type, storage, and security group at a glance.
2. When everything looks correct, click **Launch instance**.
3. You'll see a success confirmation. Click **View all instances** (or the instance ID link) to go to the Instances list.
4. Wait for the **Instance State** to change to `Running` and **Status Checks** to pass before attempting to connect.

> **Note:** There is no separate "Review and Launch" step in the current wizard; the summary panel on the right serves that purpose throughout.

### Windows Instance (RDP)

1. Select your running Windows instance.
2. Click **Connect**, then choose the **RDP client** tab.
3. Click **Get password**:
   - Upload your `.pem` private key file.
   - Click **Decrypt password** to reveal the Administrator password.
4. Download the **Remote Desktop File** and open it, or manually enter the public IP in your RDP client.
5. Log in with username `Administrator` and the decrypted password.
6. Accept any certificate warnings and proceed.

> **Note:** Windows 10 and 11 include the **Remote Desktop Connection** client built-in. On macOS, use **Microsoft Remote Desktop** from the App Store.

## Security, and Billing

### Security

- Never share your private key (`.pem` or `.ppk`).
- Keep security group rules tight; restrict SSH and RDP to known IP addresses only.
- Remove unused key pairs and security groups to reduce your attack surface.
- Consider using **AWS Systems Manager Session Manager** as a keyless alternative to SSH/RDP.

### Billing

- EC2 instances accrue charges while in a `running` state, even if idle.
- **Stop** an instance to pause compute charges (storage costs still apply). **Terminate** it to remove the instance entirely.
- Monitor your spending in the **AWS Billing Dashboard** and set up billing alerts if needed.

---

### Cleanup

- Terminate test instances when you're done.
- Delete unneeded EBS volumes, Elastic IPs, and snapshots; these can incur ongoing charges even after an instance is terminated.

---


## Recommendations

- **Region Selection:** Always confirm your region before launching; resources are region specific and won't appear if you switch regions afterward.
- **Tags:** Use tags consistently for cost tracking and resource organization, especially in multi user or production environments.
- **SSH on Windows:** Windows 10 and 11 support OpenSSH natively, so `.pem` files work directly in PowerShell or Command Prompt; PuTTY is optional.
- **AWS Documentation:** Use the **?** icon within the console for context sensitive help, or refer to the [AWS EC2 User Guide](https://docs.aws.amazon.com/ec2/index.html) for deeper detail.
